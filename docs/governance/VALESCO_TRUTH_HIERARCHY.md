# Valesco Estimating Framework — Truth Hierarchy & Dependency Model

Version: 1.9
Status: Authoritative Governance Document
Location: /docs/governance/
Purpose: Define the order of truth, dependency direction, and allowed responsibilities for every file in the Valesco system.

0. CORE PRINCIPLE: THE ACTIVATION GATEWAY

The Valesco System v1.9 operates on a "Self-Activating" model.
Truth does not exist until the system is **Activated**.

    Activation Header (Layer 0)
           ↓
   Instructions (docs)
           ↓
   Schemas (engine)
           ↓
   Registry Definitions (engine config)
           ↓
   Core Libraries (library/core)
           ↓
   Pack (library/packs)
           ↓
   Extensions (library/extensions)
           ↓
   Proposals (workspace)
           ↓
   Runtime + Exports (workspace/outputs)

This hierarchy is absolute.

1. THE ACTIVATION HEADER (Layer 0)

Authority Level: Supreme (The "Brain")
Source: Generated by `prepare_context.bat` inside `VALESCO_FULL_CONTEXT.txt`.

Responsibility:
- Forces the AI into "Strict Architect Mode".
- Commands the AI to index the layers below.
- **If the Handshake (`*** VALESCO SYSTEM ONLINE ***`) fails, no downstream truth is valid.**

2. INSTRUCTIONS (valesco_instructions.txt)

Authority Level: Highest Static Authority
Responsibility: System behaviour, units, runtime rules, validation logic.

Defines:
- Unit whitelist
- Error definitions
- Provenance rules
- Task matching
- Waste rules
- Export structure
- Cross-file validation

Cannot depend on: any YAML file, pack, schema, extension, or proposal.
All other files must obey it.

3. JSON SCHEMAS (engine/schemas/*.json)

Authority Level: High
Responsibility: Enforce structural correctness.

Ensures:
- Required keys exist
- Property types are valid
- Units match allowed enumerations
- Shapes of arrays/objects are correct

Schemas enforce structure only. Behaviour is defined by instructions.
Must obey: Instructions
Used by: `validate.bat` (Python Tools) and the AI Architect.

4. REGISTRY DEFINITIONS (engine/config/materials_allocator.yaml)

Authority Level: High
Responsibility: Define valid categories, groups, and code generation patterns.

Defines:
- Valid Category IDs (MAT.XX)
- Valid Group IDs (MAT.XX.YY)
- Code Construction Rules (MAT.STD vs MAT.BES)

Must obey: Schemas (YAML structure), Instructions (Naming conventions)
Used by: `material_manager.bat`, Core Libraries, Validator.

5. CORE LIBRARIES

Authoritative reference datasets for estimating logic.
**Hybrid Access Rule:** These files are read by the AI "Brain" (via the Bundle) AND by the Python Tools (via physical files).

5A. Materials Library (library/core/valesco_materials.yaml)
Provides truth for: Material codes (MAT.STD...), Descriptions, Units, Rates.
Must strictly follow Allocator categories.

5B. Tasks Library (library/core/valesco_tasks.yaml)
Provides truth for: Productivity, Labour/plant requirements, Output units.
**Hard links to Material Codes.**

5C. Subcontractors Library (library/core/valesco_subcontractors.yaml)
Provides truth for: Specialist supply & lay, Subcontractor methods.

6. PACK (library/packs/valesco_pack.yaml)

Authority Level: Medium
Responsibility: Project-specific defaults and overlays.

Defines:
- Labour rates, Plant rates, Waste rates, Prelim percentages.

Must obey:
- Instructions, Schemas, Materials (cannot invent codes).
- **Cannot define a materials list (must rely on Core Library).**

7. EXTENSIONS (library/extensions/*.yaml)

Authority Level: Low-Medium
Responsibility: Additive project-level items (materials, tasks, subs).

Must:
- Add only (never override).
- Use BESPOKE prefixes (MAT.BES).
- Serve as staging before promotion into core libraries.

8. PROPOSALS (workspace/proposals/*.json)

Authority Level: Low
Responsibility: Suggestions only.

Contain:
- material_new suggestions, task_mapping suggestions.

Not truth. Not authoritative.
Must be validated → merged (`merge.bat`) → promoted → validated again.

9. RUNTIME STATE

Authority Level: None
Provides: Matching, Calculations, Provenance.
It reflects validated upstream data only.

10. DEPENDENCY RULES (Set in Stone)

(1) Activation Header ←— Absolute control. If missing, system is asleep.
(2) Instructions      ←— Zero dependency on data.
(3) Schemas           ←— Shaped by instructions.
(4) Allocator         ←— Defines registry keys.
(5) Core Libraries    ←— Obey allocator & schemas.
(6) Pack              ←— Uses libraries, schemas, instructions.
(7) Extensions        ←— Additive overlays, must obey all above.
(8) Proposals         ←— Provisional, require approval.

11. EDITING RULES

Layer            | Editing Purpose        | Restrictions
-----------------|------------------------|--------------------------
Activation       | **Automated Only** | Never edit manually. Use `prepare_context.bat`.
Instructions     | Behavioural rules      | Rare edits, versioned.
Schemas          | Structural enforcement | Must mirror instructions.
Allocator        | Category definitions   | Must preserve legacy IDs.
Materials        | Master material list   | **Scripted edits only (`material_manager.bat`).**
Tasks            | Task logic             | Must follow units + instructions.
Pack             | Project defaults       | Cannot contradict upstream.
Extensions       | Add-only items         | No overrides.
Proposals        | Suggestions            | Never authoritative.

12. THE "HYBRID CONTEXT" RULE

Valesco v1.9 enforces a "Dual-Read" protocol:
1. **The Brain:** Reads the Text Bundle (`VALESCO_FULL_CONTEXT.txt`) to understand Rules and Rates.
2. **The Tools:** Read the Physical Files (`*.yaml`) to perform Validation and Math.

**Consequence:** You must always upload **BOTH** the bundle and the loose files (handled automatically by `prepare_context.bat`).

- The governance model of the Valesco v1.9 framework.

It is authoritative and must be followed for all future development.
