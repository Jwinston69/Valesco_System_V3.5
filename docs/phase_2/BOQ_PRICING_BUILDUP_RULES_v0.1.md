# BOQ Pricing Build-Up Rules v0.1

## 1. Purpose

This document defines planned Phase 2.3 pricing build-up rules for validated client BOQ items and controlled internal resource allocations.

It is a planning document only. It does not implement pricing logic, formulas, import, export, validation, tests, runtime behaviour, architecture, engine changes, library changes, bin changes or workspace changes.

Client-provided Excel BOQ import remains the first Phase 2 priority. VALESCO-generated BOQs from tender documents remain later priority and are not implemented by this document.

## 4. Pricing Layer Separation

Planned pricing must separate:

* imported client BOQ data
* normalised VALESCO BOQ item data
* resource allocations
* controlled rate sources
* deterministic build-up calculations
* preliminaries
* risk
* allowances
* assumptions and exclusions
* estimator review status

Client-facing BOQ values and internal pricing build-up values must not be silently merged.

## 5. Required Pricing Inputs

A priced BOQ item should require:

* validated BOQ item reference
* source workbook, worksheet and row traceability where available and required
* normalised quantity
* normalised unit
* one or more controlled resource allocations
* controlled rate source for every priced resource
* explicit preliminaries basis where preliminaries apply
* explicit risk basis where risk applies
* explicit allowance basis where allowances apply
* estimator review status
* validation status

No priced total should be commercially relied upon where required inputs are missing, failed or not checked.

## 6. Cost Category Rules

Pricing build-up must keep the following categories separate:

| Category | Planning rule |
| --- | --- |
| Labour | Must identify labour resource, unit, productivity or quantity basis, and controlled rate source |
| Plant | Must identify plant resource, usage basis, unit, quantity and controlled rate source |
| Materials | Must identify material resource, quantity basis, waste basis where applicable, unit and controlled rate source |
| Subcontract | Must identify subcontract scope, quote or approved basis, exclusions and controlled source reference |
| Preliminaries | Must be visible and separately based; must not be hidden inside labour, plant or margin |
| Risk | Must be visible, justified and reviewed; must not be hidden inside another category |
| Allowances | Must be visible, provisional where applicable, and linked to an assumption or unresolved item |

Future implementation may define formulas, but must preserve this separation.

## 7. Controlled Rate Source Gate

A controlled rate source is required before pricing readiness.

A resource must not become pricing-ready where its rate is:

* blank
* guessed
* copied from an unapproved workbook formula
* inferred from client return values
* generated by AI assistance without governed review
* based on an uncontrolled supplier quotation
* based on an undocumented estimator assumption
* marked `not_checked`

Where no controlled source exists, the resource status must remain `source_required`, `review_required` or `blocked` according to future validation rules.

## 8. Deterministic Calculation Requirements

Future pricing implementation must be deterministic once controlled inputs are accepted.

At planning level, deterministic calculation means:

* the same controlled inputs produce the same priced outputs
* formulas are visible, reviewable and governed
* category subtotals are reproducible
* risk and allowances are explicit
* source references remain linked to calculated outputs
* no hidden AI, workbook or runtime decision changes priced values

This document does not define executable formulas.

## 9. Build-Up Traceability

Each priced output should retain traceability to:

* BOQ item identifier
* original client BOQ source reference where applicable
* resource allocation identifier
* rate source identifier
* calculation basis
* assumption reference where applicable
* exclusion reference where applicable
* estimator review record where applicable
* validation status

If traceability cannot be preserved, the affected item must not be pricing-ready.

## 10. Review and Validation Statuses

Planned pricing build-up statuses:

| Status | Meaning |
| --- | --- |
| `not_started` | Pricing build-up has not started |
| `draft` | Pricing build-up exists but controls are incomplete |
| `not_checked` | Required checks have not run and cannot be treated as pass |
| `warning` | Pricing may require estimator review before continuation |
| `fail` | Pricing must stop for the affected scope |
| `source_required` | Controlled rate or basis is missing |
| `review_required` | Estimator review is required before reliance |
| `pricing_ready` | Required planned controls are satisfied, subject to future implementation |
| `excluded` | Item or resource is excluded with a reason |

`not_checked` must not be treated as `pricing_ready`.

## 11. Fail-Closed Pricing Rules

Future implementation must fail closed where:

* BOQ item validation is failed or not checked
* required source traceability is missing
* required quantity or unit is invalid
* resource allocation is missing for a priced BOQ item
* resource category is unknown
* controlled rate source is missing
* assumptions are missing, ambiguous or unreviewed
* risk or allowance is hidden
* subcontract scope or quote basis is uncontrolled
* estimator review is required but incomplete
* AI assistance has suggested pricing without governed review

No silent continuation, default rate, hidden provisional sum, hidden uplift or automatic acceptance of workbook formulas is permitted.

## 12. Client Return Separation

The internal pricing build-up must remain separate from any client return worksheet or client-facing tender output.

Client return values must not become internal pricing authority unless separately governed and validated.

The source client BOQ must remain preserved as evidence and must not be overwritten by pricing outputs.

## 13. Sensitive Data Controls

This document contains no real client BOQs, client-identifying data, tender returns, commercial rates, supplier quotations, project-specific quantities or project-specific sensitive information.

Future examples must be artificial, sanitised or approved before being committed.

## 14. Acceptance Criteria

These pricing build-up rules are acceptable when:

* client-provided Excel BOQ import remains first priority
* VALESCO-generated BOQ from tender documents remains later priority
* BOQ item, resource allocation and pricing build-up layers are separated
* labour, plant, materials, subcontract, preliminaries, risk and allowances are separated
* controlled rate source is required before pricing readiness
* traceability back to BOQ item and resource source is required
* fail-closed behaviour is preserved
* implemented, tested, planned, archived and unknown states are distinguished
* no functionality is claimed as implemented
* no sensitive client data is introduced
